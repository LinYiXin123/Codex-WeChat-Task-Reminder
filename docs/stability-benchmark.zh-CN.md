# 7420 稳定性对标整改清单

更新时间：2026-04-26

## 对标对象

| 项目 | 参考价值 | 对 7420 的启发 |
| --- | --- | --- |
| OpenCode | 热门、本地 agent、client/server 架构、可远程驱动 | 7420 应把服务端状态作为唯一事实源，安卓和 Web 只做薄客户端 |
| OpenAI Codex | 官方行为基线 | 队列、任务状态、权限、恢复逻辑应尽量贴近官方客户端 |
| OpenHands | 自托管 Web GUI、REST API、前端状态完整 | 参考任务生命周期、Web 恢复、前端错误降级 |
| Cline / Roo Code | IDE 内交互成熟 | 参考工具权限、上下文管理、任务状态提示 |
| aider | 成熟终端型，层级少 | 参考低复杂度状态机，减少 UI 对后端的高频干扰 |

## 当前主要不稳定源

1. RPC 过载
   - 日志中 `thread/list` 和 `thread/read` 经常需要 15 秒以上。
   - `account/rateLimits/read` 也出现 17-22 秒耗时。
   - 这些非交互 RPC 会占用 app-server 通道，导致 `turn/start` 和恢复同步被拖慢。

2. 前端轮询过密
   - 运行中任务曾以 2.5 秒甚至 1.2 秒节奏触发状态同步。
   - 当单个 RPC 已经超过 15 秒时，高频轮询只会制造排队和假卡顿。

3. Android 恢复链路还不够“快照优先”
   - 锁屏/后台后浏览器无法持续实时接收事件。
   - 回前台时应先显示本地快照，再做轻量追平，最后按需读完整消息。

4. UI 状态缺少强兜底
   - “思考中”不应只依赖单次 RPC 结果。
   - 需要用事件流、线程版本、pending request、最后活跃时间共同判断，避免任务已结束但 UI 不清理。

5. 公网/安卓入口与本地进程耦合过强
   - 本地健康、公网 FRP/隧道健康、Android WebView 会话状态需要分层检测。
   - 不能只用一个“连接正常”表达所有层都正常。

## 已纳入当前修复方向

1. `turn/start`、`thread/resume` 等交互 RPC 使用更长前端超时。
2. app-server RPC 增加队列和优先级，交互请求优先。
3. Android resume 降低重读次数，减少锁屏回来瞬间的 RPC 风暴。
4. 证书和密码输出已从公开代码/日志路径收敛。
5. 当前轮新增降载：
   - 后台同步间隔从 2.5 秒提高到 5 秒。
   - 活跃详情同步间隔从 2.5 秒提高到 6 秒。
   - 活跃 boost 间隔从 1.2 秒提高到 2.5 秒。
   - 活跃线程列表刷新从 12 秒提高到 30 秒。
   - rate-limit 最短刷新间隔限制为 120 秒。

## 下一阶段优先级

P0：服务端稳定

1. 给 `/codex-api/health` 增加最近慢 RPC 摘要：最近 5 分钟慢方法、排队峰值、最后一次 timeout。
2. 给 `thread/list` 做服务端短 TTL 缓存；事件流通知线程变化时失效。
3. 给 `account/rateLimits/read` 做服务端 TTL 缓存；失败时返回上次快照并标记 stale。
4. 把 `turn/start` 彻底从普通读请求排队中隔离，避免慢读影响发送。

P1：前端状态机

1. 线程切换先渲染缓存，再后台刷新。
2. “思考中”增加最大空闲兜底：长时间无事件、无 pending request、线程版本未变化时自动进入恢复读。
3. 刷新按钮直接触发强制同步，并显示当前同步结果，不再静默失败。
4. 错误提示区分：本地服务、app-server、公网转发、登录态、权限请求。

P1：Android

1. 回前台先做轻量快照恢复，不立即全量读消息。
2. WebView 登录态持久化和 cookie/localStorage 检查单独做健康项。
3. 输入框聚焦时只调整 composer 区域，不能隐藏顶部菜单或侧边栏。
4. 折叠屏使用双栏专属布局，不沿用普通手机高度逻辑。

P2：产品化

1. 保留 7420 作为 Codex Web/Android 薄客户端，不继续扩成重型 agent runtime。
2. 若后续要彻底稳定，优先参考 OpenCode 的 client/server 协议，而不是继续堆前端补丁。
3. GitHub 发行版只保留 `2.1.x` 这类版本号，不再使用英文后缀。
